# Архитектура системы (Lab 2)

## Назначение системы
Система предназначена для предварительной оценки риска диабета 2 типа на основе короткой анкеты пользователя.
Результат используется как ориентировочная информация и не является диагнозом.

## Общая архитектура решения
В Lab 2 используется простая двухуровневая архитектура:
- frontend (клиентская веб-страница с анкетой)
- backend (REST API для расчета балла риска)

Архитектура намеренно упрощена для учебного MVP: без базы данных, без авторизации и без внешних интеграций.

## Основные компоненты системы
| Компонент | Назначение | Технология (MVP) |
|---|---|---|
| Frontend | Сбор данных анкеты, отправка запроса, показ результата | HTML/CSS/JavaScript |
| Backend API | Прием анкеты, валидация, расчет риска, формирование ответа | FastAPI |
| Risk Logic Module | Правила начисления баллов и определение уровня риска | Python (rule-based) |

## Роль frontend
- Отображает анкету и пояснение к системе.
- Выполняет базовую проверку полей (числовые диапазоны, обязательные значения).
- Отправляет данные на backend и показывает результат: `score`, `risk_level`, `recommendation`.

## Роль backend
- Предоставляет endpoint `GET /health` для проверки доступности сервиса.
- Предоставляет endpoint `POST /risk-assessment` для расчета риска.
- Выполняет серверную валидацию входных данных.
- Возвращает результат в формате JSON.

## Где находится логика оценки риска
Логика оценки риска вынесена в отдельный модуль backend:
- `backend/app/services/risk_logic.py`

Это позволяет менять правила расчета без изменения маршрутов API и упрощает последующий переход к Lab 3.

## Поток данных от пользователя до результата
1. Пользователь открывает страницу frontend и заполняет анкету.
2. Frontend отправляет JSON-запрос в `POST /risk-assessment`.
3. Backend валидирует данные и передает их в модуль risk logic.
4. Модуль вычисляет балл и категорию риска.
5. Backend возвращает JSON-ответ с рекомендацией.
6. Frontend отображает результат пользователю.

## Вариант локального развертывания
- Backend запускается локально на `http://localhost:8000`.
- Frontend запускается как статическая страница на `http://localhost:5500`.
- Взаимодействие идет по HTTP в локальной среде разработки.

## Почему архитектура упрощена для этапа MVP
- Цель Lab 2: подготовить понятный и демонстрируемый прототип.
- Упрощение снижает объем реализации и риски для студенческой команды.
- Базовая архитектура покрывает ключевой сценарий из Lab 1: заполнение анкеты и получение предварительной оценки риска.
- Расширения (история оценок, более точные модели, интеграции) перенесены на последующие этапы.
