# Архитектура системы (Lab 2)

## Назначение системы
Система предназначена для предварительной оценки риска диабета 2 типа на основе короткой анкеты пользователя.
Результат используется как ориентировочная информация и не является диагнозом.

## Общая архитектура решения
Используется простая двухуровневая архитектура:
- frontend (веб-страница с формой анкеты и отображением результата);
- backend (REST API с валидацией и rule-based расчетом).

Архитектура намеренно упрощена для учебного MVP: без базы данных, авторизации и внешних интеграций.

## Основные компоненты системы
| Компонент | Назначение | Технология |
|---|---|---|
| Frontend | Сбор данных анкеты, отправка запроса, показ результата | HTML/CSS/JavaScript |
| Backend API | Прием анкеты, валидация, расчет риска, формирование ответа | FastAPI |
| Risk Logic Module | Начисление баллов и определение категории риска | Python (rule-based) |

## Роль frontend
- Отображает анкету и дисклеймер.
- Выполняет базовую клиентскую проверку значений.
- Отправляет данные на backend.
- Отображает `score`, `risk_level`, `recommendation`, `explanation`.

## Роль backend
- Предоставляет endpoint `GET /health`.
- Предоставляет endpoint `POST /risk-assessment`.
- Валидирует входные данные.
- Возвращает результат в формате JSON.

## Где находится логика оценки риска
Логика вынесена в модуль:
- `backend/app/services/risk_logic.py`

Такой подход упрощает поддержку и делает расчет прозрачным для учебной защиты.

## Поток данных от пользователя до результата
1. Пользователь открывает frontend и заполняет анкету.
2. Frontend отправляет JSON в `POST /risk-assessment`.
3. Backend валидирует данные и передает их в `risk_logic.py`.
4. Модуль расчета формирует балл и категорию риска.
5. Backend возвращает `score`, `risk_level`, `recommendation`, `explanation`.
6. Frontend показывает результат пользователю.

## Вариант локального развертывания
- Backend: `http://localhost:8000`
- Frontend: `http://localhost:5500`
- Взаимодействие: HTTP-запросы в локальной среде.

## Почему архитектура упрощена
- Цель Lab 2 — подготовить реалистичный и демонстрируемый каркас MVP.
- Простая структура снижает трудозатраты и риск ошибок в студенческом проекте.
- Сложные функции (история, админ-режим, интеграции) перенесены на будущие этапы.
