# Дизайн-заметки (Lab 2)

## Контекст
В Lab 1 зафиксированы требования к учебной системе предварительной оценки риска диабета 2 типа.
В Lab 2 выполнен переход к проектным решениям и структуре прототипа, который затем реализован в Lab 3.

## Основные проектные решения
1. Использовать двухуровневую архитектуру: frontend + backend API.
2. Реализовать расчет риска через понятные rule-based правила.
3. Обрабатывать данные в рамках одного запроса без хранения в БД.
4. Оставить интерфейс одностраничным и ориентированным на один главный сценарий.

## Почему выбрана легковесная архитектура
- Проект учебный и ограничен по времени.
- Важнее показать целостный рабочий поток, чем сложную инфраструктуру.
- Простая структура облегчает объяснение решения на защите.

## Допущения для MVP
- Пользователь вводит корректные базовые данные.
- Оценка является предварительной и ориентировочной.
- Рекомендации формируются шаблонно, без клинической персонализации.

## Ключевые элементы UI
- Заголовок и краткое описание системы.
- Форма анкеты с базовыми факторами риска.
- Кнопка расчета.
- Блок результата: `score`, `risk_level`, `recommendation`, `explanation`.
- Явный дисклеймер о недиагностическом характере результата.

## Границы backend-модулей
- `main.py` — API-роуты и конфигурация приложения.
- `schemas.py` — модели входа/выхода и валидация.
- `risk_logic.py` — расчет баллов и интерпретация результата.

## Принятые упрощения
- Нет авторизации и личного кабинета.
- Нет хранения истории оценок.
- Нет внешних интеграций.
- Нет ML-обучения на медицинских датасетах.

## Ограничения текущего подхода
- Rule-based модель не заменяет клиническую оценку.
- Охват факторов риска ограничен выбранной анкетой.
- Изменение правил требует ручной корректировки кода.
