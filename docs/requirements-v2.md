# Обновленные требования (Requirements v2, Lab 3)

## Цель обновления требований
Цель документа — зафиксировать уточнение требований после перехода к MVP в Lab 3.
Обновление выполнено на основе материалов Lab 1 и Lab 2 с учетом ограничений учебного формата и необходимости демонстрации рабочего прототипа.

## Что выбрано для MVP
В MVP включены только функции, необходимые для демонстрации основного сценария:
- веб-форма анкеты пользователя;
- базовая клиентская и серверная валидация;
- расчет итогового балла риска по прозрачным правилам;
- определение уровня риска (низкий/средний/высокий);
- вывод краткой рекомендации и пояснения к результату;
- endpoint проверки доступности сервиса (`GET /health`).

## Что намеренно отложено
Следующие функции исключены из Lab 3 и перенесены на будущие этапы:
- регистрация/авторизация пользователей;
- хранение истории результатов;
- административная панель;
- интеграции с внешними медицинскими системами;
- использование клинически валидированных ML-моделей.

## Ключевые реализованные пользовательские сценарии
1. Пользователь открывает страницу с описанием и дисклеймером.
2. Пользователь заполняет анкету и отправляет данные.
3. Система вычисляет балл риска и категорию.
4. Пользователь получает результат и рекомендацию.
5. Пользователь может изменить данные анкеты и выполнить повторную оценку.

## Упрощенные нефункциональные требования
- Локальный запуск проекта должен выполняться стандартными командами без сложной инфраструктуры.
- Время ответа API на расчет — в пределах нескольких секунд при демонстрационной нагрузке.
- Интерфейс должен быть понятным для пользователя без специальной подготовки.
- Код и структура проекта должны быть читаемыми для учебной проверки.

## Ограничения MVP
- Используется rule-based логика, а не клиническая модель.
- Расчет не учитывает полный набор медицинских показателей.
- Результаты носят ориентировочный характер и не являются диагнозом.

## Критерии приемки MVP
MVP считается принятым, если:
1. Backend успешно запускается локально и отвечает на `GET /health`.
2. Frontend запускается локально и отправляет анкету в backend.
3. Для корректного запроса backend возвращает JSON с `score`, `risk_level`, `recommendation`, `explanation`.
4. Пользователь видит результат расчета на странице frontend.
5. В интерфейсе и документации присутствует явный дисклеймер о недиагностическом характере системы.
