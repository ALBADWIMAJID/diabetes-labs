# Отчет по интеграции (Lab 3)

## Цель интеграции
Проверить связность frontend и backend в рамках MVP-сценария предварительной оценки риска диабета 2 типа.

## Статус backend
- Backend реализован на FastAPI.
- Доступны endpoints:
  - `GET /health`
  - `POST /risk-assessment`
- Входные данные валидируются через Pydantic.
- Расчет реализован на основе rule-based логики.

## Статус frontend
- Реализована одна страница с формой анкеты.
- Добавлена базовая клиентская проверка полей.
- Настроена отправка POST-запроса в backend.
- Реализован вывод результата в отдельном блоке.

## Статус интеграции
Интеграция выполнена успешно для основного пользовательского сценария.
Frontend и backend взаимодействуют по HTTP в локальной среде разработки.

## Что работает end-to-end
1. Открытие страницы frontend.
2. Ввод данных анкеты.
3. Отправка запроса в `POST /risk-assessment`.
4. Получение ответа backend (`score`, `risk_level`, `recommendation`, `explanation`).
5. Отображение результата и дисклеймера в интерфейсе.

## Что замокано или упрощено
- Нет постоянного хранения данных (без БД).
- Нет авторизации и личного кабинета.
- Нет клинической валидации алгоритма.
- Логика оценки представлена в виде простых правил для учебного MVP.

## Выявленные проблемы при интеграции
1. Нужна была единая структура ответа для удобного отображения в UI.
2. Требовалась более понятная обратная связь при сетевых ошибках.

## Решения и отложенные пункты
- Добавлено поле `explanation` в ответ backend для объяснения результата.
- Во frontend улучшена обработка ошибок и сообщений пользователю.
- Отложено на следующие этапы: расширенный набор тестов и более детальная персонализация рекомендаций.

## Рекомендации по развитию
- Добавить небольшой набор автотестов для API.
- Вынести конфигурацию адреса API во внешний конфиг frontend.
- Сравнить rule-based baseline с простым ML-подходом на валидированном датасете в учебных целях.
